# Plan 2026-06-08

Current baseline: `v0.1.206` / commit `436b96e`.

## What the timeline says is done

- Video recording is shipped as a fixed five-second MP4 (`29.0`, `29.2`, `29.3`).
- Ears are baked into recorded video through CameraX `OverlayEffect` (`fac433b`, hardened by
  `2f17a92` and `436b96e`).
- Recorded-video ghosting/drift from reused overlay canvases is fixed (`2f17a92`).
- Recorded-video crop/rotation mapping now uses CameraX frame crop metadata (`436b96e`).
- Lens switching during active recording is blocked and redundant camera rebinds are skipped
  (`e690b88`).
- Party Mode uses stable per-tracking-ID style assignments and avoids replacement-face slot
  collisions (`f15432a`).
- User-facing tint cycling and non-photorealistic procedural styles were removed from the app
  surface (`4645c0f`).

## Coding Plan

### P0 - Release confidence

1. Build a local debug APK and run a real-device smoke test.
   Acceptance: preview opens, front/rear switch works, photo saves, five-second video saves with ears,
   Party Mode re-rolls styles for multiple faces, landscape placement is acceptable.
2. Capture current screenshots and one short demo clip for README/release assets.
   Acceptance: media committed under `media/`, README embeds current realistic ears and Party Mode.
3. Verify GitHub Actions after push.
   Acceptance: CI, CodeQL, dependency review, secret scan, and release workflow status are known.

### P1 - Bugfixes

1. Fix exception-path bitmap cleanup in `OverlayCompositor.composite`.
   Status: done in `HEAD`; the compositor recycles its allocated mutable copy before rethrowing
   draw failures, preserving the caller's plain-frame fallback without abandoning native bitmap
   memory.
2. Persist help dialog language selection with `rememberSaveable`.
   Current gap: rotating/recreating the help dialog resets the selected language to English.
   Acceptance: selected help language survives Compose state recreation; focused UI/domain test if
   practical; full CI passes.
3. Add a defensive no-face video overlay test strategy.
   Current gap: the real behavior depends on CameraX `OverlayEffect`; host tests cannot instantiate
   it directly.
   Acceptance: extract the pure frame mapping/overlay-state logic enough to unit-test empty
   placements, crop mapping, and front-camera mirroring decisions.

### P1 - Feature Gaps

1. Animated GIF export from the five-second MP4.
   Scope: add "Export GIF" beside Share Video, downsample to a small social-friendly format, and
   save/share through MediaStore.
   Acceptance: GIF export is optional, failure-safe, and does not block normal video sharing.
2. README screenshots and demo media.
   Scope: replace text-only feature claims with visual proof of realistic ears, video, and Party
   Mode.
   Acceptance: README first viewport includes current screenshots/demo media.
3. First-launch onboarding.
   Scope: a compact, dismissible first-run guide showing capture, style cycle, video, and Party
   Mode controls.
   Acceptance: persisted dismissal, no marketing landing page, accessible controls.

### P2 - Testing and Validation Gaps

1. Add screenshot tests for core Compose states.
   Target states: permission required, camera error, ready controls, recording, saved photo/video,
   help dialog languages.
2. Add instrumented happy-path tests.
   Target: permission grant, preview shell, capture state transitions, share intents.
3. Add emulator E2E in CI when the runner/toolchain is stable enough.
   Target: Gradle Managed Device or documented fallback script.
4. Resolve Robolectric/JDK blocker or configure a JDK 21 test toolchain.
   Target: host-side Android tests for lifecycle/camera-adjacent code without waiting on JDK 26 ASM
   support.

### P2 - Technical Debt

1. Consolidate live/capture/video renderer geometry.
   Current debt: `CatEarOverlay` and `OverlayCompositor` still duplicate renderer decisions and
   fallback geometry.
   Target: shared pure geometry/spec layer plus thin Compose/Canvas adapters.
2. Bound sprite bitmap caches and make them lifecycle-aware.
   Current debt: decoded sprites live for the process lifetime.
   Target: bounded cache or Android resource/cache helper with a clear memory-pressure story.
3. Decide the future of `EarTint` / `EarTintPolicy`.
   Current debt: tint metadata remains in domain because Party Mode/history used it, but the shipped
   UI no longer exposes tinting.
   Target: either remove tint metadata from active paths or document it as reserved for a future
   sprite-safe tint feature.
4. Update Android 10 downport spike after current release.
   Target: run `:app:checkDebugAarMetadata` with `minSdk=29`, document dependency blockers, then
   test on an API 29 emulator and one real Android 10 device.

## Recommended Order

1. P0 real-device smoke test and debug APK.
2. P1 help-language `rememberSaveable`.
4. README screenshots/demo media.
5. Renderer/cache/tint debt cleanup.
6. GIF export or onboarding, depending on release feedback.
