## The Good — 5 things done right

### 1. Clean domain module with zero Android deps

`:domain` is a pure JVM module. All geometry (`imageToViewCoordinates`, `computeOverlayPlacement`),
state machines (`CaptureState`, `RecordingState`), and smoothing logic (`MultiFaceSmoother`) run
without Robolectric. 1 189 test lines execute in milliseconds on any JVM. This is not an accident —
it was a deliberate architectural choice enforced by the module boundary, and it shows.

### 2. Sealed-class state machines enforce exhaustive handling

`CaptureState`, `RecordingState`, `MainUiState`, and `PermissionState` are all sealed hierarchies.
The `when` expressions in `MainScreen` and `MainActivity` are exhaustive by the compiler. No stringly-
typed status flags, no boolean soup. Adding a new state is a compile error until every call site
handles it — exactly what you want.

### 3. Per-tracking-ID face smoother is memory-safe

`MultiFaceSmoother` keyed `PlacementSmoother` instances on ML Kit face-tracking IDs and calls
`smoothers.keys.retainAll(activeIds)` each frame. Smoothers for faces that leave the frame are
discarded immediately; memory is bounded by the number of simultaneously visible faces. The EMA
alpha=0.3 value is also well-chosen — responsive but not jittery. Small thing, done correctly.

### 4. Correct scoped-storage with `IS_PENDING`

`ImageSaver` sets `IS_PENDING = 1` before writing and clears it only after the stream is flushed.
If the write fails the incomplete entry is deleted from MediaStore. No legacy `READ_EXTERNAL_STORAGE`
permission, no `WRITE_EXTERNAL_STORAGE`. This is the right way to write to the gallery on API 29+
and it is easy to get wrong.

### 5. Coordinate transform is well-modelled and tested

`imageToViewCoordinates` applies scale, crop offset, and front-camera horizontal mirror as distinct,
named steps. It is backed by 108 lines of tests in `CoordinateTransformTest.kt` covering scale,
aspect ratio, mirroring, and edge cases. The function is readable by anyone who has not touched the
code before. Most AR projects bury this logic in a 100-line anonymous blob — this one externalised
it properly.

## The Bad — 10 open issues (updated 2026-06-04, pass 2)

### 4. Front-camera video overlay is horizontally mirrored

### 1. `RecordingState.Failed` is a dead-end with no recovery path

`MainViewModel.onRecordTap()` is gated by `if (_recordingState.value is RecordingState.Idle)`.
Once a recording fails (`onRecordingSaved(null)` → `RecordingState.Failed`), this guard permanently
blocks the record button: `RecordButton.isRecording = recordingState is RecordingState.Recording`
evaluates to false, so the FAB calls `onRecordTap()` which no-ops. `onRecordingConsumed()` is
only wired from `MainActivity` when the state is `Saved`, never `Failed`.

The user has no way to recover without force-closing the app. No toast, no caption, no retry
button is shown for the failed state. Fix: handle `Failed` the same as `Idle` in `onRecordTap()`,
or show a dismissible error that calls `onRecordingConsumed()` to reset.

### 2. `isFrontCamera` captured by value in `AndroidView.factory` closure

`configure()` is called from the `factory` block of `AndroidView`, which runs exactly once.
The `onFaceResult` lambda inside `CameraBindConfig` captures the `lens` parameter from the first
composition:

```kotlin
onFaceResult = { faces, w, h ->
    val transform = TransformContext(
        isFrontCamera = lens == LensSelector.Front,  // ← captured at factory time
        ...
    )
```

After the user switches lenses, `AndroidView.update` fires `controller.bindPreview(newLens)`
(updating the video overlay's `isFrontCamera` field correctly), but the live-preview
`TransformContext.isFrontCamera` remains the value from the initial composition. The face
coordinate transform uses the wrong mirror assumption → ears appear on the wrong sides of the
face in the live preview after every lens switch.

Fix: pass the current `lens` as a LaunchedEffect dependency so `configure()` is re-called when
it changes, or read the current lens from an `AtomicReference` inside the lambda.

### 3. `ImageProxy` close contract is inconsistent between the interface and its implementation

`FaceDetectorSeam` declares: "@param imageProxy The current camera frame (caller must close it
when done)." Under this contract, `analyseFrame` in `CameraXControllerImpl` should close the
proxy after calling `detector.process()`. It only does so in the catch block.

`MlKitFaceDetectorImpl` actually closes the proxy itself via `addOnCompleteListener { imageProxy.close() }`,
which works because ML Kit's pipeline is asynchronous — closing the proxy in the caller would race
the detector. The two halves of the system have silently agreed on the opposite of what the
interface says. A future seam implementation that follows the documented contract would leak every
camera frame.

Fix: update the `FaceDetectorSeam` docstring to state "the implementation is responsible for
closing the proxy," and remove the catch-block close from `analyseFrame`.

### 4. `MlKitFaceDetectorImpl` KDoc lies about its face-selection strategy

The class comment says "Selects the face with the largest bounding box when multiple are detected."
The implementation does:

```kotlin
onResult(faces.take(FaceDetectorSeam.MAX_FACES).map { it.toFaceModel() })
```

`take` returns faces in ML Kit's internal return order, which is not documented as size-ordered.
In group selfies the wrong face often gets the cat ears. The "largest face" behaviour described in
the docs was either never implemented or was removed without updating the comment.

Fix: sort by `(boundingBox.width() * boundingBox.height()).toDouble()` descending before `take`.

### 5. `MainActivity` injects `MlKitFaceDetectorImpl` directly, bypassing `FaceDetectorSeam`

```kotlin
@Inject lateinit var faceDetectorProvider: Provider<MlKitFaceDetectorImpl>  // line 44
```

The `FaceDetectorSeam` interface exists precisely to decouple ML Kit from the rest of the app
and to enable test doubles. But the DI wiring at the only real injection site ignores it.
Swapping the detector (A/B test, ML Kit update, on-device model) requires changing `MainActivity`
rather than a Hilt binding.

Fix: inject `Provider<FaceDetectorSeam>` and bind `MlKitFaceDetectorImpl` to the seam in a Hilt
module with `@Binds`.

### 6. `saveLayer(null, ...)` in the tint path allocates a full-frame offscreen buffer per face

In `OverlayCompositor.drawEarsOnCanvas()` and `composite()`, tinted placements call:
```kotlin
canvas.saveLayer(null, Paint().apply { colorFilter = filter })
```
`null` bounds means "clip to the current clip rect" — for the video overlay draw listener,
that is the entire video frame (e.g., 1920 × 1080 ARGB_8888 = 8 MB). With four tinted faces
at 30 fps, ~240 MB of offscreen surfaces are allocated per second, and GC pressure causes
dropped video frames.

Fix: pass tight bounds covering only the two ear anchors instead of `null`, and cache the
`Paint` + `ColorMatrixColorFilter` per tint value (only 6 possible values).

### 7. `frame.getOverlayCanvas()` called twice inside the `withMatrix` block

`CameraXControllerImpl.buildOverlayEffect()`, lines 142-143:
```kotlin
frame.getOverlayCanvas().withMatrix(matrix) {
    OverlayCompositor.drawEarsOnCanvas(frame.getOverlayCanvas(), state.placements)
}
```
Inside `withMatrix`, `this` is already the matrix-transformed canvas. The code ignores `this`
and calls `frame.getOverlayCanvas()` a second time, relying on the fact that both calls return
the same object. This is correct today but fragile: if `getOverlayCanvas()` ever returns a
fresh canvas per call, ears would be drawn at untransformed coordinates.

Fix: `OverlayCompositor.drawEarsOnCanvas(this, state.placements)`.

### 8. `previewView.width` / `previewView.height` read on the analysis background thread

Inside `CameraBindConfig.onFaceResult` (set in the `AndroidView.factory` block,
`CameraPreviewComposable.kt` lines 118-119):
```kotlin
viewWidth = previewView.width.coerceAtLeast(1),
viewHeight = previewView.height.coerceAtLeast(1),
```
This callback runs on `analysisExecutor`. `View.getWidth()` reads an int field written by the
main thread without synchronization. On ARM with a weak memory model, a background thread can
observe a stale zero (before the first layout pass), producing a `TransformContext` with
`viewWidth = 1`, which collapses all ear coordinates to the top-left corner.

Fix: cache the dimensions in an `AtomicInteger` pair updated from the main thread via
`addOnLayoutChangeListener`.

### 9. `hueRotationMatrix()` allocates three objects on the draw hot path per tinted face

Every call to `hueRotationMatrix(hueDegrees)` allocates a `FloatArray(20)`. In
`OverlayCompositor.drawEarsOnCanvas()` and `composite()`, three more objects are created
around it: `ColorMatrix(floatArray)`, `ColorMatrixColorFilter(matrix)`, and `Paint()`.
At 4 tinted faces × 30 fps, that is 480 short-lived objects per second in the camera draw
thread, triggering GC pauses that manifest as frame drops in video recording.

There are only 6 `EarTint` values. Fix: precompute and cache one `Paint` per tint as a lazy
companion property; skip allocation entirely in the hot path.

### 10. `videoCapture` field not nulled after `unbind()`

`buildVideoUseCase()` sets `videoCapture = VideoCapture.withOutput(recorder)` (line 161-165).
`unbind()` calls `cameraProvider?.unbindAll()` but does not null out `videoCapture`. If
`startVideoRecording()` is called in the window between `unbind()` and the next
`bindPreview()` — possible during the lens-switch sequence — it finds `videoCapture != null`,
calls `vc.output.prepareRecording(...)` on a detached use case, and either throws an uncaught
exception or silently creates a corrupt MediaStore entry.

Fix: `videoCapture = null` at the top of `unbind()`.
