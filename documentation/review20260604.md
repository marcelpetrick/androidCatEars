Scope: app source code and architecture. Not CI/CD, not documentation, not build tooling.
Reviewer: Claude Code (Sonnet 4.6) on behalf of Marcel Petrick.

## The Good — 5 things done right

### 1. Clean domain module with zero Android deps

`:domain` is a pure JVM module. All geometry (`imageToViewCoordinates`, `computeOverlayPlacement`),
state machines (`CaptureState`, `RecordingState`), and smoothing logic (`MultiFaceSmoother`) run
without Robolectric. 1 189 test lines execute in milliseconds on any JVM. This is not an accident —
it was a deliberate architectural choice enforced by the module boundary, and it shows.

### 2. Sealed-class state machines enforce exhaustive handling

`CaptureState`, `RecordingState`, `MainUiState`, and `PermissionState` are all sealed hierarchies.
The `when` expressions in `MainScreen` and `MainActivity` are exhaustive by the compiler. No stringly-
typed status flags, no boolean soup. Adding a new state is a compile error until every call site
handles it — exactly what you want.

### 3. Per-tracking-ID face smoother is memory-safe

`MultiFaceSmoother` keyed `PlacementSmoother` instances on ML Kit face-tracking IDs and calls
`smoothers.keys.retainAll(activeIds)` each frame. Smoothers for faces that leave the frame are
discarded immediately; memory is bounded by the number of simultaneously visible faces. The EMA
alpha=0.3 value is also well-chosen — responsive but not jittery. Small thing, done correctly.

### 4. Correct scoped-storage with `IS_PENDING`

`ImageSaver` sets `IS_PENDING = 1` before writing and clears it only after the stream is flushed.
If the write fails the incomplete entry is deleted from MediaStore. No legacy `READ_EXTERNAL_STORAGE`
permission, no `WRITE_EXTERNAL_STORAGE`. This is the right way to write to the gallery on API 29+
and it is easy to get wrong.

### 5. Coordinate transform is well-modelled and tested

`imageToViewCoordinates` applies scale, crop offset, and front-camera horizontal mirror as distinct,
named steps. It is backed by 108 lines of tests in `CoordinateTransformTest.kt` covering scale,
aspect ratio, mirroring, and edge cases. The function is readable by anyone who has not touched the
code before. Most AR projects bury this logic in a 100-line anonymous blob — this one externalised
it properly.

## The Bad — open issues (updated 2026-06-04, pass 3)

Reviewer: Claude Code (Sonnet 4.6). Scope: post-fix codebase at v0.1.170.
Severity scale: **CRITICAL** (data loss / crash in normal use) · **HIGH** (active feature bug, hits regularly) ·
**MEDIUM** (degraded experience, reproducible) · **LOW** (edge case or cosmetic).

---

### 1. Party-mode slot collision after pruning — HIGH

**Status: fixed on 2026-06-05.** Party mode now assigns replacement faces from a monotonic
per-group slot counter and resets the counter only after all tracked faces disappear. Regression
tests cover the documented `[2,4]` replacement case.

`MainViewModel.partyFaceSlots` is correctly pruned by `retainAll(activeIds)` on each
`onFaceDetected()` call, but the slot-assignment formula `partyFaceSlots.getOrPut(id) { partyFaceSlots.size }`
uses `size` *at insertion time*. After pruning, the map is sparse; `size` is the count of
surviving entries, which can equal an already-occupied slot value.

Concrete example:
```
Frame 1:  faces [1,2,3] arrive → partyFaceSlots = {1→0, 2→1, 3→2}
Frame 2:  faces [2,4] — prune removes 1,3 → {2→1}
          face 4 inserted: size = 1 → slot 1
          → partyFaceSlots = {2→1, 4→1}   ← collision!
```
Both faces now have slot 1 → identical `earStyle` and `tint`. Party mode shows the same ears on
every face whenever a new face replaces an old one. The fix for unbounded growth (pass 1) introduced
this regression; the correct approach is a monotonically increasing counter separate from map size.

**Severity: HIGH** — reproduces every time a face enters after another has left. Core feature
broken for groups.

### 2. Captured frame bitmap (`previewView.bitmap`) never recycled — MEDIUM

**Status: fixed on 2026-06-05 in `f15432a`.** The ViewModel recycles the saved bitmap after
MediaStore save completion, and the capture compositor recycles the original preview frame after
successfully producing a separate composited bitmap.

`captureComposited()` calls `previewView?.bitmap` which creates a new `Bitmap` backed by native
memory (typically 5–30 MB on modern sensors). The function either returns it directly (no face) or
passes it to `OverlayCompositor.composite()` which creates *another* copy. The original `frame`
variable is abandoned in both paths without calling `frame.recycle()`.

```kotlin
val frame = previewView?.bitmap          // allocates native memory
...
return compositeEarsOrFrame(frame, placements)   // frame passed into composite(), then abandoned
```

On burst-capture or repeated photos, unreleased bitmaps pile up in native heap faster than the GC
can collect them, leading to `OutOfMemoryError` on low-RAM devices.

**Severity: MEDIUM** — silent memory leak on every photo capture; crash risk on low-memory devices.

### 3. `MultiFaceSmoother` not reset between lens switches — MEDIUM

**Status: fixed on 2026-06-05 in `e690b88`.** Lens changes reset `MultiFaceSmoother` and clear
current overlay placements before the next camera result arrives.

`val smoother = remember { MultiFaceSmoother() }` is created once and survives recompositions,
including lens switches. After switching from front to rear (or vice versa), the smoother still
holds EMA state from the previous camera — face positions that are in a completely different
coordinate space.

On the first frame(s) from the new camera where a face *is* detected, the smoother interpolates
from the old (wrong) position toward the new one, causing ears to visibly slide in from an off-screen
anchor rather than appearing immediately above the face.

`smoother.reset()` is called when `faces.isEmpty()`, but if the new camera finds a face immediately
there may be zero empty frames between the switch and the first detection.

Fix: reset the smoother whenever `lens` changes. This can be done with `LaunchedEffect(lens) { smoother.reset() }`.

**Severity: MEDIUM** — reproducible visual artifact after every lens switch when a face is in frame.

### 4. Lens switch while recording silently truncates the clip — MEDIUM

**Status: fixed on 2026-06-05 in `e690b88`.** The ViewModel ignores lens toggles while recording,
and `CameraXControllerImpl` defensively refuses camera rebinds while a recording is active.

`CameraPreview`'s `AndroidView.update` block calls `controller.bindPreview(newLens)` whenever
`lens` changes. Inside `bindPreview()`, `provider.unbindAll()` detaches all use cases including
`VideoCapture`. CameraX then fires a `VideoRecordEvent.Finalize` event asynchronously, which
triggers `onFinished(uri)` and saves whatever was recorded so far.

The user has no indication that toggling the lens stopped the recording mid-clip. There is no
warning, no confirmation dialog, and no distinct recording-truncated state. The saved clip is
simply shorter than expected.

Fix: check `activeRecording != null` at the top of `bindPreview()` and either refuse the switch
while recording, or call `stopVideoRecording()` first and notify the user.

**Severity: MEDIUM** — data-loss adjacent; a user who switches lenses mid-recording gets a partial
clip with no feedback about why it is short.

### 5. `compositeEarsOrFrame` leaks the bitmap copy on draw exception — LOW

**Status: still a low-priority follow-up.** The normal capture path now recycles owned bitmaps, but
an exception thrown inside `OverlayCompositor.composite()` after allocating its mutable copy could
still abandon that copy until GC. This is exception-path-only and belongs in the next hardening pass.

`OverlayCompositor.composite()` creates a full-resolution mutable copy of the frame bitmap at the
very first line (`result = frame.copy(...)`). If any of the ear-drawing calls throw *after* this
allocation, the exception escapes `composite()`. `compositeEarsOrFrame` catches it and returns the
original `frame` — but `result` is abandoned without `recycle()`, leaking native heap until the
next GC cycle.

This only happens on the exception path (a genuine draw failure), so its practical frequency is
low. But the fix is one line: `result.recycle()` in the catch block.

**Severity: LOW** — exception-path-only leak; one large bitmap (~10–30 MB) released at next GC.

### 6. `@Volatile isFrontCamera` — one-frame mirroring flicker on lens switch — LOW

**Status: fixed on 2026-06-04 in `9fd358c`.** Front-camera video mirroring is corrected in the
overlay transform path.

`isFrontCamera` is set to the new value in `bindPreview()` (main thread) before the new camera
is bound. The `OverlayEffect` draw listener runs on `overlayEffectThread` and reads `isFrontCamera`
to decide whether to apply the horizontal flip. Between the write and the next memory-fence on the
draw thread, there is a window of one or two video frames where the draw listener may still see the
old value — ears mirrored on the wrong side for those frames.

`@Volatile` guarantees that the write is eventually visible but does not prevent this brief window.
A practical fix: write `isFrontCamera` after the new `OverlayEffect` is created (i.e., after
`buildOverlayEffect()` returns) so the new draw listener only ever sees the correct value.

**Severity: LOW** — cosmetic, one or two frames at most, visible only to a careful observer.

---

### 7. Drawing logic still fully duplicated across two renderers — MEDIUM

Ten ear styles are implemented independently in `OverlayCompositor.kt` (Android `Canvas`) and
`CatEarOverlay.kt` (Compose `DrawScope`). The visual fixes in pass 1 reduced divergence, but the
structural duplication remains: adding or changing any ear style still requires editing both files.
The two implementations will drift silently again.

The correct fix — a pure geometry layer producing path/shape descriptions consumed by thin Canvas
and DrawScope adapters — was identified in pass 1 and scoped as "larger than one PR." It remains
the most significant architectural debt in the drawing subsystem.

**Severity: MEDIUM** — no user-visible bug today, but the next ear style addition will produce a
divergence within one commit.
