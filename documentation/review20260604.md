# Code Review — androidCatEars — 2026-06-04

Scope: app source code and architecture. Not CI/CD, not documentation, not build tooling.
Reviewer: Claude Code (Sonnet 4.6) on behalf of Marcel Petrick.

---

## The Good — 5 things done right

### 1. Clean domain module with zero Android deps

`:domain` is a pure JVM module. All geometry (`imageToViewCoordinates`, `computeOverlayPlacement`),
state machines (`CaptureState`, `RecordingState`), and smoothing logic (`MultiFaceSmoother`) run
without Robolectric. 1 189 test lines execute in milliseconds on any JVM. This is not an accident —
it was a deliberate architectural choice enforced by the module boundary, and it shows.

### 2. Sealed-class state machines enforce exhaustive handling

`CaptureState`, `RecordingState`, `MainUiState`, and `PermissionState` are all sealed hierarchies.
The `when` expressions in `MainScreen` and `MainActivity` are exhaustive by the compiler. No stringly-
typed status flags, no boolean soup. Adding a new state is a compile error until every call site
handles it — exactly what you want.

### 3. Per-tracking-ID face smoother is memory-safe

`MultiFaceSmoother` keyed `PlacementSmoother` instances on ML Kit face-tracking IDs and calls
`smoothers.keys.retainAll(activeIds)` each frame. Smoothers for faces that leave the frame are
discarded immediately; memory is bounded by the number of simultaneously visible faces. The EMA
alpha=0.3 value is also well-chosen — responsive but not jittery. Small thing, done correctly.

### 4. Correct scoped-storage with `IS_PENDING`

`ImageSaver` sets `IS_PENDING = 1` before writing and clears it only after the stream is flushed.
If the write fails the incomplete entry is deleted from MediaStore. No legacy `READ_EXTERNAL_STORAGE`
permission, no `WRITE_EXTERNAL_STORAGE`. This is the right way to write to the gallery on API 29+
and it is easy to get wrong.

### 5. Coordinate transform is well-modelled and tested

`imageToViewCoordinates` applies scale, crop offset, and front-camera horizontal mirror as distinct,
named steps. It is backed by 108 lines of tests in `CoordinateTransformTest.kt` covering scale,
aspect ratio, mirroring, and edge cases. The function is readable by anyone who has not touched the
code before. Most AR projects bury this logic in a 100-line anonymous blob — this one externalised
it properly.

## The Bad — 10 open issues (updated 2026-06-04)

### 1. `partyFaceSlots` grows without bound and corrupts slot assignments

`MainViewModel.partyFaceSlots` (`LinkedHashMap<Int, Int>`, line 54) accumulates face tracking IDs
forever. ML Kit tracking IDs are monotonically increasing — each new detected face gets a fresh ID.
After a typical session (panning the phone, showing photos, detecting many different faces),
the map can hold hundreds of stale entries.

The slot number for a new face is `partyFaceSlots.size` at insertion time. As the map grows, new
faces are always assigned ever-higher slot indices. Once `slot ≥ PARTY_STYLE_ORDER.size`, the
`shuffledIndex` formula wraps with integer arithmetic that was designed for `slot < 10`. The result
is unpredictable style assignment, not the round-robin the feature promises.

`MultiFaceSmoother` correctly prunes dead tracking IDs each frame via `retainAll`; the ViewModel
slot map never does. The fix: mirror `MultiFaceSmoother`'s pruning pattern — after `onFaceDetected`,
call `partyFaceSlots.keys.retainAll(activeTrackingIds)` to evict stale entries.

### 2. `OverlayEffect` instance leaked on every lens switch

`CameraXControllerImpl.buildOverlayEffect()` creates a new `OverlayEffect` each time `bindPreview`
is called (line 92: `val effect = buildOverlayEffect()`). `OverlayEffect` implements `AutoCloseable`
and internally starts an OpenGL processing pipeline (`SurfaceProcessorImpl`). The old instance is
never `close()`d — neither in `unbind()` nor in `close()`. Every front/rear toggle leaves an
unreleased GPU texture pipeline and a running background thread.

Fix: store the current `OverlayEffect` in a field; call `currentEffect?.close()` before creating
the next one, and call it again in `close()`.

### 3. `Handler.postDelayed` auto-stop timer is not cancellable

```kotlin
Handler(Looper.getMainLooper()).postDelayed({ stopVideoRecording() }, VIDEO_DURATION_MS)
```
The `Runnable` is created at line 195 and immediately orphaned — no reference is stored so it can
never be cancelled. If the user stops the recording early at 2 s and starts a new one at 3 s, the
old timer fires at 5 s and calls `stopVideoRecording()` in the middle of the second recording,
silently truncating it. Fix: store the `Runnable`, cancel it in `stopVideoRecording()` and `close()`.

### 4. Front-camera video overlay is horizontally mirrored

`viewToBufferMatrix` (`CameraXControllerImpl.kt`, companion object) correctly maps view-space
coordinates to the buffer given rotation, but ignores `Frame.isMirroring()`. For the front camera,
the view shows a selfie-mirrored image (left/right flipped) while the buffer is not mirrored. The
ears are therefore drawn with their x-coordinates reflecting the mirrored view, placing them on the
wrong sides of the face in the recorded video — left ear appears on the right, and vice versa.

Fix: after building the rotation matrix, prepend a horizontal flip when `frame.isMirroring()` is
`false` but the view coordinates are known to be mirrored (i.e., front lens):
```kotlin
if (!frame.isMirroring() && isFrontCamera) matrix.preScale(-1f, 1f, fW / 2f, fH / 2f)
```
`isFrontCamera` must be threaded from `VideoOverlayState` (add it to the data class and set it
in `updateOverlayPlacements`).

### 5. Drawing logic duplicated across two files

Ten ear styles are implemented independently in `OverlayCompositor.kt` (Android `Canvas`) and
`CatEarOverlay.kt` (Compose `DrawScope`). They have already diverged: `CANINE_PERKY` uses an
`arcTo` rounded cap in the Compose overlay but a `drawCircle` in the compositor; `DENSE_FLUFFY`
has animated fur strands only in the live overlay; the compositor version is static. Every new
ear style requires two implementations. The only correct fix is a single geometry layer that
both renderers consume — a pure-geometry model that produces vertex/path data, then thin adapters
for Canvas and DrawScope.

### 6. `CameraXControllerImpl` is a 7-property mutable bag

The class has accumulated seven `var` properties that must be wired by the caller before
`bindPreview()` is safe: `previewView`, `lifecycleOwner`, `context`, `faceDetector`,
`onFaceResult`, `onBindFailed`, plus `updateOverlayPlacements()` as a mutation method. Calling
`bindPreview()` before any of them are set either silently returns (null guard) or crashes.
This is property injection in a project that otherwise uses Hilt constructor injection. The class
was added to the review's original list as a 6-property bag; `onBindFailed` was added during bug
fixes and made it worse, not better. Extract a value-object parameter type and pass it to
`bindPreview` as a single argument.

### 7. No Compose UI tests

The test suite has 37 ViewModel tests and 17 domain/seam test files — zero `createComposeRule()`
tests. None of the following is caught by CI:
- Party mode toggle showing the correct FAB state
- `HelpDialog` opening and language switching working
- Share FAB appearing only after `CaptureState.Saved`
- `CameraError` screen rendering with a retry button
- `RecordButton` showing the stop icon during recording

The Stop-button bug (WP 35) lived undetected for the entire feature implementation. A single
`createComposeRule()` test would have caught it on the first run. Compose UI tests can run
on the JVM with `@get:Rule val rule = createComposeRule()` — no device required.

### 8. `startVideoRecording` has no concurrent-start guard

`CameraXControllerImpl.startVideoRecording()` (line 176) sets `activeRecording` without checking
whether a recording is already in progress. If called twice in rapid succession — possible when a
`LaunchedEffect` fires twice before the state settles — the first `Recording` handle is overwritten
and leaked. The underlying CameraX `Recording` will keep writing to the first MediaStore entry, its
`Finalize` callback will fire but `onFinished` won't be called (the reference is gone), and the
ViewModel stays in `RecordingState.Recording` forever.

Fix: `if (activeRecording != null) return` at the top of the function.

### 9. `AppModule.provideApplicationContext` is redundant

```kotlin
@Provides @Singleton
fun provideApplicationContext(@ApplicationContext context: Context): Context = context
```
Hilt already provides `@ApplicationContext Context` with a qualifier at every injection site.
This binding re-provides it as an unqualified `Context`, which hides which context you actually
get. Any future injection of plain `Context` will silently resolve to the Application context even
when Activity context would be more correct. Delete it and use `@ApplicationContext Context`
explicitly at every injection site (`ImageSaver`, etc.).

### 10. `CaptureRuntime` lives in the wrong package

`CaptureRuntime` and `DefaultCaptureRuntime` are defined in `it.marcelpetrick.catears.ui`
(file: `CaptureRuntime.kt`). This is an I/O abstraction — it owns the coroutine dispatcher,
the wall-clock, and a random-suffix generator — that exists solely to make `MainViewModel`
unit-testable. It belongs in `capture` or a top-level `runtime` package. A developer reading
the `ui` package expects composables and ViewModels, not infrastructure seams.
