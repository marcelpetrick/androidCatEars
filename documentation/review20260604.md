# Code Review — androidCatEars — 2026-06-04

Scope: app source code and architecture. Not CI/CD, not documentation, not build tooling.
Reviewer: Claude Code (Sonnet 4.6) on behalf of Marcel Petrick.

---

## The Good — 5 things done right

### 1. Clean domain module with zero Android deps

`:domain` is a pure JVM module. All geometry (`imageToViewCoordinates`, `computeOverlayPlacement`),
state machines (`CaptureState`, `RecordingState`), and smoothing logic (`MultiFaceSmoother`) run
without Robolectric. 1 189 test lines execute in milliseconds on any JVM. This is not an accident —
it was a deliberate architectural choice enforced by the module boundary, and it shows.

### 2. Sealed-class state machines enforce exhaustive handling

`CaptureState`, `RecordingState`, `MainUiState`, and `PermissionState` are all sealed hierarchies.
The `when` expressions in `MainScreen` and `MainActivity` are exhaustive by the compiler. No stringly-
typed status flags, no boolean soup. Adding a new state is a compile error until every call site
handles it — exactly what you want.

### 3. Per-tracking-ID face smoother is memory-safe

`MultiFaceSmoother` keyed `PlacementSmoother` instances on ML Kit face-tracking IDs and calls
`smoothers.keys.retainAll(activeIds)` each frame. Smoothers for faces that leave the frame are
discarded immediately; memory is bounded by the number of simultaneously visible faces. The EMA
alpha=0.3 value is also well-chosen — responsive but not jittery. Small thing, done correctly.

### 4. Correct scoped-storage with `IS_PENDING`

`ImageSaver` sets `IS_PENDING = 1` before writing and clears it only after the stream is flushed.
If the write fails the incomplete entry is deleted from MediaStore. No legacy `READ_EXTERNAL_STORAGE`
permission, no `WRITE_EXTERNAL_STORAGE`. This is the right way to write to the gallery on API 29+
and it is easy to get wrong.

### 5. Coordinate transform is well-modelled and tested

`imageToViewCoordinates` applies scale, crop offset, and front-camera horizontal mirror as distinct,
named steps. It is backed by 108 lines of tests in `CoordinateTransformTest.kt` covering scale,
aspect ratio, mirroring, and edge cases. The function is readable by anyone who has not touched the
code before. Most AR projects bury this logic in a 100-line anonymous blob — this one externalised
it properly.

## The Bad — 10 things to fix or be embarrassed about

### 1. `CameraXControllerImpl` is a mutable property bag

Six `var` properties — `context`, `previewView`, `lifecycleOwner`, `faceDetector`,
`onFaceResult`, and (newly) `videoOverlayState` — must be wired from the caller in the right order
before `bindPreview()` is safe. This is property injection in a project that already uses Hilt
constructor injection everywhere else. If any of these are set late, or in the wrong order, or not
at all, `bindPreview()` silently returns (null guard) or crashes. The class exists outside the Hilt
graph even though it is `@Inject constructor()`. Extract a proper factory or pass dependencies into
`bindPreview`.

### 2. Drawing logic is duplicated across two files

10 ear styles are implemented **twice**: once in `OverlayCompositor.kt` using
`android.graphics.Canvas`, once in `CatEarOverlay.kt` using Compose `DrawScope`. Any geometry
or colour change must be applied to both. They have already diverged — the `CANINE_PERKY` style
uses an `arcTo` arc cap in `CatEarOverlay` but a `drawCircle` hack in `OverlayCompositor`. The
`DENSE_FLUFFY` fur animation exists only in the live overlay; the captured version is static.
This is a latent consistency bug that will grow with every new ear style. The long-term fix is a
single renderer that accepts either a Canvas or a DrawScope via an adapter.

### 3. The Stop button doesn't stop recording

`onRecordTap()` in `MainViewModel`:
```kotlin
fun onRecordTap() {
    if (_recordingState.value is RecordingState.Idle) {
        _recordingState.value = RecordingState.Recording
    }
}
```
The guard is `is Idle`. During recording the state is `Recording`. Tapping the red Stop FAB fires
`onRecordTap()` but the guard blocks it — **nothing happens**. The user sees a Stop icon but the
button is a lie. The clip always runs the full five seconds regardless of the user's intent. This
needs a separate `onStopRecording()` that calls `controller.stopVideoRecording()` and transitions
the state.

### 4. `PlacementSmoother` drops `trackingId` on every frame

`PlacementSmoother.smooth()` returns an `OverlayPlacement` constructed without `trackingId`:
```kotlin
val smoothed = OverlayPlacement(
    leftEar = smoothAnchor(...),
    rightEar = smoothAnchor(...),
    headEulerAngleY = lerp(...),
    smilingProbability = lerp(...),
    eyeOpennessMean = lerp(...),
    // trackingId: not set → null
)
```
`CatEarOverlay` then does `key(placement.trackingId ?: index)`. Since `trackingId` is always null
post-smoothing, it always falls back to index. When two faces swap positions in the detection
results, the Compose keys change, which restarts the spring animations and causes ear-flickering
artefacts. The fix is one line: propagate `trackingId = next.trackingId` in the smoothed output.

### 5. No cancellation on the 5-second video timer

```kotlin
Handler(Looper.getMainLooper()).postDelayed({ stopVideoRecording() }, VIDEO_DURATION_MS)
```
The `Handler` and `Runnable` are created and immediately forgotten — there is no reference saved to
cancel them. If the composable is disposed (app backgrounds, camera closed, activity rotated) before
5 seconds elapse, `stopVideoRecording()` fires on a dead controller. The `activeRecording` field
will already be null so nothing crashes, but the `onFinished` callback may never be delivered,
leaving the ViewModel permanently in `RecordingState.Recording`. Save the `Runnable` and cancel it
in `close()`.

### 6. `CaptureRuntime` is in the wrong package

`CaptureRuntime` and `DefaultCaptureRuntime` live in `it.marcelpetrick.catears.ui`. This is an I/O
abstraction interface (controls the dispatcher, clock, and random suffix) that exists solely to make
`MainViewModel` testable. It belongs in `capture` or a dedicated `runtime` package, not `ui`. A
new developer reading the UI package will be confused by what this class is doing there.

### 7. `AppModule.provideApplicationContext` is redundant and confusing

```kotlin
@Provides @Singleton
fun provideApplicationContext(@ApplicationContext context: Context): Context = context
```
This injects the Hilt-qualified `@ApplicationContext Context` and re-provides it as an unqualified
`Context` singleton. Hilt already provides `@ApplicationContext Context` and `@ActivityContext
Context` with proper qualifiers. This unqualified provision hides which context you actually get at
each injection point and could cause subtle bugs if someone later provides a different `Context` that
conflicts. Delete it — use `@ApplicationContext Context` at each injection site instead.

### 8. No Compose UI tests whatsoever

The test suite has 1 189 lines across 17 test files. All of them test either domain logic or the
ViewModel in isolation. Zero tests cover:
- whether `HelpDialog` opens when the info button is tapped
- whether the Share FAB appears only after a capture succeeds
- whether the Record button changes to a Stop icon while recording
- whether `CameraContent` vs `PermissionRequiredContent` is rendered correctly

The wiring between state and composables is completely untested. A Compose test that runs on the
`@HiltAndroidTest` device test infrastructure, or even a simple `createComposeRule()` unit test
with a fake ViewModel, would catch the Stop-button bug (item 3 above) immediately.

### 9. Video overlay coordinate transform ignores buffer orientation

The `OverlayEffect` `onDraw` callback scales placement coordinates as:
```kotlin
val sx = frame.getSize().width.toFloat() / state.viewWidth
val sy = frame.getSize().height.toFloat() / state.viewHeight
```
CameraX `VideoCapture` with `Quality.HIGHEST` typically stores frames in sensor-landscape space
(e.g., 1920 × 1080) with a rotation tag in the MP4 container, while the app runs in portrait. This
means `frame.getSize()` returns a landscape `Size` while `state.viewWidth/Height` is portrait. The
scale factors will be inverted relative to the display orientation, placing ears in completely wrong
positions. The fix: check `frame.getRotationDegrees()` and rotate/swap dimensions accordingly.

### 10. Camera bind failures are silently swallowed

```kotlin
@Suppress("TooGenericExceptionCaught")
try {
    provider.unbindAll()
    provider.bindToLifecycle(owner, selector, groupBuilder.build())
    Log.d(TAG, "Camera bound (lens=$lens)")
} catch (e: Exception) {
    Log.e(TAG, "Camera bind failed", e)
}
```
If binding fails (camera already in use by another app, unsupported use-case combination,
`IllegalArgumentException` from CameraX), the composable renders a black screen and the logcat
receives a red line that most users will never see. There is no state transition to tell the ViewModel
something went wrong, no snackbar, no retry. For a camera app, silent failure is the worst possible
outcome. The ViewModel should have a `CameraError` state and the composable should expose a retry
button.
